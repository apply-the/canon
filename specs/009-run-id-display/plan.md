# Implementation Plan: Run Identity, Display Id, and Authored-Input Refactor

**Branch**: `009-run-id-display` | **Date**: 2026-04-22 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/009-run-id-display/spec.md`

## Summary

Introduce a clean two-name model for Canon runs: keep the existing UUIDv7 as
the immutable canonical machine identity (`uuid`), and add a stable
human-facing display id `run_id = R-YYYYMMDD-SHORTID` (UTC date + first 8
hex chars of the UUID), with optional descriptive `slug` and `title`
metadata. Refactor the run directory layout to
`.canon/runs/YYYY/MM/R-YYYYMMDD-SHORTID[--slug]/`, formalize the contract
between the editable `canon-input/` authoring surface and the immutable
per-run snapshot under `.canon/runs/<…>/inputs/`, and extend CLI run
resolution to accept `run_id`, unique `short_id`, legacy `uuid`, and
optionally `@last`. Read compatibility for legacy UUID-keyed run directories
is preserved without forced migration.

## Governance Context

**Execution Mode**: change  
**Risk Classification**: bounded-impact — touches identity, persistence
layout, CLI resolution, and authored-input handling across the engine and
CLI; preserves the orchestrator, governance gates, skill system, and
external dependency surface unchanged. No database, no protocol changes, no
silent destructive moves.  
**Scope In**: run manifest schema additions; deterministic display-id and
slug derivation; new dated on-disk layout; authored-input vs run-snapshot
contract enforcement; CLI lookup by `run_id`/`short_id`/`uuid` (and
optionally `@last`); run listing surface; docs/help text alignment;
read-compatibility shim for legacy UUID-keyed run directories.  
**Scope Out**: replacing UUID; introducing a numeric counter as identity;
new persistence engine; redesigning gates, zones, risk, skills, or
orchestrator; moving `.canon/`; bulk silent migration; non-identity UX work.

**Invariants**:

- UUID (UUIDv7) remains the immutable canonical machine identity; nothing
  derived from authored content or slugs may replace it.
- Files under `canon-input/` MUST NOT be mutated by runtime persistence;
  per-run snapshots under `.canon/runs/<…>/inputs/` MUST remain immutable
  evidence with preserved digests and provenance.
- `status`, `inspect evidence`, `inspect artifacts`, `approve`, `resume`,
  and gate / verification / invocation persistence MUST keep working for
  both new and legacy runs.
- Slug and title are descriptive metadata only; never identity, uniqueness,
  or lookup correctness.
- Display-id parsing relies on the first `--` separator only; later `--`
  occurrences belong to the slug payload.

**Decision Log**: [decision-log.md](decision-log.md), cross-linked from the
change run's `.canon/decisions/` records.  
**Validation Ownership**: Implementation in `crates/canon-engine` and
`crates/canon-cli` is generated by the implementer; validation is performed
by `cargo nextest run`, `cargo fmt --check`, `cargo clippy … -D warnings`,
the skill validator scripts, contract tests under `tests/contract/`, and an
independent human reviewer (not the implementer) performing the steps in
[quickstart.md](quickstart.md). Validation evidence lives in
[validation-report.md](validation-report.md).  
**Approval Gates**: bounded-impact requires (a) green structural and
logical validation, (b) documented decisions in `decision-log.md`, and
(c) one independent human reviewer sign-off on the merge before the change
ships. Systemic-impact escalation is not expected; if discovered during
implementation, the plan must be re-classified before proceeding.

## Technical Context

**Language/Version**: Rust 1.95.0, Edition 2024  
**Primary Dependencies**: `clap`, `serde`, `serde_json`, `serde_yaml`,
`toml`, `thiserror`, `tracing`, `uuid` (v7), `time`, `sha2`  
**Storage**: Local filesystem under `.canon/` only; TOML manifests, Markdown
artifacts. No new persistence engine.  
**Testing**: `cargo test` / `cargo nextest run`; existing contract,
integration, and snapshot tests under `tests/`.  
**Target Platform**: macOS, Linux, Windows operator workstations and CI
runners; the installed `canon` binary is the daily-use surface.  
**Project Type**: Single-language Rust workspace (CLI + engine + adapters).  
**Existing System Touchpoints**:
- `crates/canon-engine/src/persistence/layout.rs` (`run_dir`, `runs_dir`,
  `run_inputs_dir`, `run_artifact_dir`, `run_evidence_path`, etc.)
- `crates/canon-engine/src/persistence/store.rs` (run write/read, evidence
  links, snapshot fingerprint paths)
- `crates/canon-engine/src/persistence/manifests.rs` (run manifest TOML)
- `crates/canon-engine/src/orchestrator/service.rs` (UUID generation,
  RunRequest/RunContext threading)
- `crates/canon-engine/src/domain/run.rs` (run record types)
- `crates/canon-cli/src/commands/{run,status,inspect,approve,resume}.rs`
  (run resolution and listing)
- Documentation: `MODE_GUIDE.md`, `README.md`, embedded skill help text  
**Performance Goals**: No regression on run create / load latency;
listing 1000 runs MUST complete in well under one second on a developer
laptop.  
**Constraints**: No new dependency. No new external service. No bulk
filesystem migration. Read compatibility for legacy UUID-keyed run
directories.  
**Scale/Scope**: Single-repo `.canon/` directory, expected up to a few
thousand run directories per repo over time.

existing `tests/contract/`, `tests/integration/`, and `tests/fixtures/`
trees. No new top-level directories are introduced.
